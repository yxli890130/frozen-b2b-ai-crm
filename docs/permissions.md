# 角色权限设计

第一版以单用户演示为主，代码层先不接完整登录权限。权限设计先作为企业级项目表达材料保留，第二阶段再接 Supabase Auth 和 RLS。

## 角色

| 角色 | 主要目标 |
| --- | --- |
| 销售 | 管理自己的客户和跟进记录 |
| 销售主管 | 查看团队客户，识别高潜线索 |
| 产品经理 | 查看客户反馈和未成交原因 |
| 管理员 | 管理字段、账号、权限和系统配置 |

## 权限矩阵

| 功能 | 销售 | 销售主管 | 产品经理 | 管理员 |
| --- | --- | --- | --- | --- |
| 查看客户列表 | 自己客户 | 团队客户 | 脱敏查看 | 全部 |
| 新增客户 | 可用 | 可用 | 不可用 | 可用 |
| 编辑客户 | 自己客户 | 团队客户 | 不可用 | 全部 |
| 删除客户 | 不可用 | 审批后 | 不可用 | 可用 |
| 添加沟通记录 | 自己客户 | 团队客户 | 不可用 | 可用 |
| 生成 AI 分析 | 可用 | 可用 | 可用 | 可用 |
| 查看 AI 分析 | 自己客户 | 团队客户 | 脱敏查看 | 全部 |

## 安全原则

- API Key 只保存在服务端环境变量。
- AI 结果是建议，不自动覆盖客户原始信息。
- 删除、关键字段修改需要确认。
- 后续通过 Supabase RLS 控制数据访问边界。
